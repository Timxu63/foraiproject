# AI 需求澄清问答入口设计

日期：2026-06-22
项目：`D:\foraiproject`
关联目标：让本 Unity 工程成为对策划、美术等非工程用户友好、面向 AI 的工程。

## 目标

建立第一版“AI 需求澄清问答入口”。它不是要求策划或美术一次性写出完整需求文档，而是让 AI 像需求分析助手一样，在现有 `Workflow Engine` 内通过一轮轮简单问题把模糊想法整理成结构化需求。

第一期只建设文档层设计和后续实施范围，不创建 Unity Editor 窗口，不新增 CLI 子命令，不修改 Unity 资产。

## 当前问题

现有工程已经具备较清晰的 AI 安全执行骨架：

- `AGENTS.md` 定义了 Agent 边界和 workflow 入口。
- `docs/ai/architecture.md`、`workflows.md`、`risk-policy.md` 等说明了工程层规则。
- `tools/ai/ai.py` 能创建 workflow、扫描 context、校验 artifact、做 risk review 和 preflight。
- Unity 感知型修改必须通过 `Unity Editor Adapter`。

这些内容对 Agent 和工程人员有用，但对策划、美术仍然不够友好。问题不在于他们不会填表，而在于他们通常无法提前知道哪些信息对工程实现必要。要求他们自己整理完整模板，仍然会产生门槛。

因此第一期入口要从“用户填模板”调整为“AI 主动问清楚”。

## 设计原则

1. 用户只需要表达初始意图，例如“我想做一个背包界面”或“帮我整理这批图标”。
2. AI 负责识别任务类型，并按任务类型逐步提问。
3. 每次只问少量问题，优先使用可选项和短回答。
4. 用户可以回答“不知道”，AI 必须给出安全默认建议或说明为什么必须确认。
5. AI 在进入修改前，必须把问答结果汇总成用户可审阅的需求摘要。
6. 问答结果只能作为 `intent-analysis`、`requirement-check`、`domain-spec` 和 `execution-plan` 的输入，不能绕过 `risk review`、`preflight` 或 Unity Adapter。
7. 涉及场景、Prefab、ProjectSettings、Package manifest、批量移动或删除时，必须明确提示风险并等待用户确认。

## 第一阶段文档范围

后续实施应新增或更新以下文档。

### `docs/ai/user-guide.md`

面向策划和美术，说明如何与 AI 协作。重点不是教用户写完整需求，而是让用户知道：

- 可以直接用自然语言描述想法。
- AI 会根据任务类型继续追问。
- 用户可以回答“不知道”。
- AI 在修改前会汇总需求、风险和验收方式。
- 用户需要重点确认目标、素材、路径、是否允许覆盖、验收结果。
- AI 不能直接乱改 Unity 资产，所有 Unity 感知型变更必须经过项目规定的执行层。

该文档应避免工程术语堆叠。必须出现工程边界时，使用短句解释。

### `docs/ai/request-templates.md`

面向 Agent 和用户共同使用。它不只是静态填空模板，而是分类型“提问脚本”。

每个任务类型包含：

- 适用场景。
- AI 的识别线索。
- 推荐提问顺序。
- 用户回答“不知道”时的默认处理。
- 需求摘要格式。
- 继续进入计划或修改执行阶段前必须确认的风险点。

第一期至少覆盖这些任务类型：

- UI 页面新增或修改。
- Prefab 创建或修改。
- 美术资源导入或整理。
- 数值或配置调整。
- Bug 复现和修复请求。
- 只读解释或检查请求。

### `docs/ai/acceptance-checklists.md`

面向用户验收，也采用引导式问题。它不只列清单，而是让 AI 带用户确认结果：

- 你能否在 Unity 或运行画面中看到目标对象？
- 文案、布局、资源、数量、交互是否符合你的预期？
- AI 是否列出了实际改动范围？
- 是否通过了必要验证？
- 是否存在你没有授权的覆盖、移动或删除？
- 你接受本次结果，还是需要继续调整？

验收清单应根据任务类型区分。例如 UI 关注显示和交互，资源导入关注路径、命名和引用，Bug 修复关注复现步骤和回归验证。

## 问答流程

标准流程如下：

1. 用户给出一句自然语言意图。
2. AI 判断任务类型；如果无法判断，先问“这是新增、修改、修复、整理，还是只想了解现状？”
3. AI 使用对应任务类型的提问脚本，每轮问一到三个短问题。
4. AI 对“不知道”的回答给出默认建议，或标记为必须确认的问题。
5. AI 汇总需求摘要，内容包括目标、对象、路径、输入素材、期望行为、限制、验收标准和风险点。
6. 需求澄清和摘要记录在对应的 `question` 或 `auto` workflow 内完成。
7. 用户确认需求摘要。
8. AI 继续进入 `plan`、`change` profile 或修改执行阶段，并生成或登记后续 artifact。
9. 修改型任务必须完成 `risk review` 和 `preflight` 后才能执行。
10. 执行后 AI 使用 `acceptance-checklists.md` 带用户验收。

## UI 任务提问脚本示例

适用意图：

- “做一个背包界面”
- “改一下登录弹窗”
- “加一个活动入口”

推荐提问：

1. 这是新界面，还是修改已有界面？
2. 这个界面什么时候打开，给玩家完成什么事情？
3. 页面上需要显示哪些信息？
4. 有哪些按钮或可点击区域？点击后分别发生什么？
5. 有没有参考图、现有页面、Prefab 或素材路径？
6. 是否允许创建新的脚本、Prefab 或 Addressables 条目？
7. 你希望怎么验收这个页面？

如果用户回答“不知道”：

- 对路径不知道：AI 根据现有项目结构建议路径，但在摘要中标记为“建议路径，待确认”。
- 对交互不知道：AI 先建议只做静态展示，不实现业务行为。
- 对是否允许创建 Prefab 不知道：默认不覆盖已有 Prefab，只提出计划。

需求摘要示例：

```text
任务类型：UI 页面新增
目标：新增背包页面，用于查看道具列表。
建议路径：Assets/_Project/HotUpdate/Features/Inventory/UI
显示内容：道具图标、名称、数量。
交互：点击道具后显示详情；关闭按钮返回上一层。
风险：可能创建新 UI 脚本和 Prefab，执行前需要确认。
验收：页面能打开，列表能显示测试数据，按钮行为符合约定。
```

## Prefab 任务提问脚本示例

推荐提问：

1. 你要创建新 Prefab，还是修改已有 Prefab？
2. Prefab 用在什么场景或系统里？
3. 目标路径或现有 Prefab 路径是什么？
4. 需要哪些组件、子物体或引用资源？
5. 是否允许覆盖已有 Prefab？
6. 是否有命名规则或参考对象？
7. 验收时要看到什么？

默认策略：

- 不知道路径时，AI 只建议路径，不直接创建。
- 不知道是否覆盖时，默认禁止覆盖。
- 涉及已有 Prefab 修改时，风险至少需要用户明确确认。

## 美术资源任务提问脚本示例

推荐提问：

1. 资源类型是什么，例如图标、立绘、音效、材质或模型？
2. 资源现在在哪里，目标放到哪里？
3. 是否需要改名？命名规则是什么？
4. 是否需要加入 Addressables 分组？
5. 是否允许移动或覆盖已有资源？
6. 导入后怎么判断结果正确？

默认策略：

- 默认不删除旧资源。
- 默认不覆盖同名资源。
- 默认不直接改 `.meta`。
- Addressables 分组不明确时，只生成计划并要求确认。

## Bug 任务提问脚本示例

推荐提问：

1. 你看到了什么异常现象？
2. 从打开项目到出现问题，需要哪些步骤？
3. 你期望看到什么结果？
4. 实际结果是什么？
5. 这个问题是否稳定复现？
6. 最近是否改过相关 UI、Prefab、配置或脚本？
7. 修复后你希望用什么方式验收？

默认策略：

- 复现步骤不完整时，AI 先帮用户补齐复现描述。
- 不能复现时，AI 先做只读排查，不直接修改。
- 修复运行时行为时，必须补充相关测试或验证命令。

## 验收问答设计

AI 执行完成后，不应只说“完成”。必须带用户完成验收问答：

1. 本次实际修改了哪些文件或资源？
2. 哪些验证已经运行，结果是什么？
3. 用户要在 Unity 或游戏中检查哪个入口？
4. 用户看到的结果是否符合需求摘要？
5. 是否有未验证或需要人工判断的部分？
6. 用户是否接受本次改动？

如果用户不接受，AI 应进入修复流程，而不是把任务标记为完成。

## 安全边界

该入口文档不能降低项目安全要求。

- `Workflow Engine` 仍只调度阶段，不直接调用 Unity API。
- `Agent` 仍只分析上下文并产出结构化结果，不直接修改 Unity 资产。
- `CLI` 仍是人、Agent、CI 和未来 MCP 工具的稳定入口。
- `Unity Editor Adapter` 仍是唯一允许创建、修改、移动或删除 Unity 资产、场景、Prefab、ProjectSettings 或编辑器状态的执行层。

用户友好不等于减少确认。它意味着 AI 要把确认问题问得更清楚。

## 验收标准

本设计规格完成后，应满足：

- 说明了从静态模板改为问答式澄清的原因。
- 定义了 `user-guide.md`、`request-templates.md` 和 `acceptance-checklists.md` 的职责。
- 给出了 UI、Prefab、美术资源、Bug 四类任务的提问脚本示例。
- 明确了“不知道”回答的默认处理方式。
- 明确后续实施仍必须遵守 workflow、risk review、preflight 和 Unity Adapter 边界。
- 文档可以正常读取，且不包含未完成标记。

## 后续实施建议

规格经用户审阅后，再写实施计划。实施计划应先创建这三个 `docs/ai` 文档，并更新 `docs/ai/project-map.md` 或 `docs/ai/workflows.md` 的文档入口。等文档问答脚本稳定后，再考虑 CLI 向导或 Unity Editor 窗口。
